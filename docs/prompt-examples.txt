
# ADD MCP TO ENGINE PLAN

plan-it

Nowadays our system is really robust but we are missing one important concept. MCPs! Nowadays our brain-connectors and brain integration are really powerfull and it is a good way to quickly integrate and to ingest data into the brain, but we don't have a easy and plug-in-play way to let LLM and brain-apps to directly use the MCP that these third-party provide us. 

/goal reasearch the ENgine codebase, how we structure module and features to create a plan on how can we leverage third-party MCPs availability to our tenants.

In my view, in the "integrations" settings, like we have backfill, webhooks, we now can add an MCP session, and if that third-party has it, we let the user install. Like we did in "automations" and "ontology", we can have an AI area where the user asks or pass the MCP documentation, github, or url and we install it for them. This would be a new session in the integration json. 

The impact of this is:
- brain-agent, zeroclaw, backbones needs to have visibility of the MCPs available and this is add to the smart-composer so the user call tag it (new type, new color, summit with "/")
- brain-apps has access to them as well. Like we give integrations so brain-apps doesn't need to handle this, the MCP needs to have the same behaviour. Also, our MCP gateway that we build for the backbones and brain-agents should also be available native to the brain-apps, so they can easily call and use brain-agent, Skills, AI Agents, and easy access to the data of our ontology.

A few MCPs examples
- https://developers.google.com/google-ads/api/docs/developer-toolkit/mcp-server
- https://developers.meta.com/horizon/essentials/ai-tooling-mcp/
- https://developers.facebook.com/documentation/mcp
- https://developers.hubspot.com/ai-tools/mcp
- https://support.airtable.com/docs/using-the-airtable-mcp-server

/plan-it

We implement the Palantir ontology in our brain, what improve A LOT our system. 
Now we need to continue the Palantir research, now focusing on a major sucess business story of their 5 days POC playbook.

"The mechanism that makes even the fast part possible is the FDE model. FDEs are full-stack software engineers who work directly inside customer environments, often for long periods, building production systems that support real operational decisions — not traditional consultants or sales engineers. This is what lets them show something working on live client data within days. Palantir popularized this so effectively that Amazon recently announced a $1 billion investment to implement forward deployed engineers in customer environments, essentially copying the playbook."

One of the most impressive things they accomplish on their system is the following:

"The kinetic half — this is what really drives the speed
Actions are where a lot of the deployment velocity comes from. An action type is a schema definition of a set of edits a user can make to objects, properties, and links all at once, plus side effects. The crucial detail: a form UI is automatically generated from the action definition, so the writeback logic and the user interface aren't defined separately. A classic example is a "Modify Flight Destination" action — you define which object and which properties it can change, and the operator-facing form comes for free."

And that is exactly what I'm aiming for the brain.
We will have hundreds of tenants, and each tenant might be from a different sector entirely. 
One might be a hospital, other an accounting firm, other a marketing agency, other a soil farm or a oil industry. How to make the customization and to fit the brain in their reality fast, is crucial for the suceess of what we are building.
